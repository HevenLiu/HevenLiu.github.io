Dubbo负载均衡策略(软负载)

前言

  • 在集群负载均衡时，Dubbo提供了多种均衡策略，缺省为random随机调用
  • 源码有部分中文注释，若不想看源码，可直接跳至本文最后的Summary
  • 负载均衡发生在哪里？

  • 可以看出Load Balance发生在Invoker，即RPC(Remote Procedure Call Protocol远程过程调用)时

  • ps:这不是Dubbo的RPC调用过程

    ​

  • Duboo支持4中负载均衡策略,详情如下

随机 Random Load Balance实现(Default)

  • 如名字，按权重设置随机概率
  • 在一个截面上碰撞的概率高，调用量越大分布越均匀，而且按概率使用权重后也比较均匀，有利于动态调整提供者权重。

轮循Round Robin Load Balance

  • 每一次把来自用户的请求轮流分配给内部中的服务器。从1开始，一直到N(N是内部服务器的总个数[N>=1])，然后重新开始循环。
  • 存在慢的提供者累积请求问题，比如：第二台机器很慢，但没挂，当请求调到第二台时就卡在那，久而久之，所有请求到第二台机器的请求都卡在第二台上，不影响其他线路。
  • 解决办法 :结合权重，把第二台机(性能低的)的权重设置低一点

最小活跃Least Active Load Balance

  • 最少活跃调用数，相同活跃数的随机，活跃数 指调用前后计数差 。

  • 使慢的提供者收到更少请求，因为越慢的提供者的调用前后计数差 会越大。

一致性哈希 Consistent Hash Load Balance

  • 一致性Hash，相同参数的请求总是发到同一提供者。
  • 当某一台提供者挂时，原本发往该提供者的请求，基于虚拟节点，平摊到其它提供者，不会引起剧烈变动。

  • 算法参见：[click Here][http://en.wikipedia.org/wiki/Consistent_hashing]

Round Robin Load Balance

  • 每一次把来自用户的请求轮流分配给内部中的服务器。从1开始，一直到N(N是内部服务器的总个数[N>=1])，然后重新开始循环。

  • 存在慢的提供者累积请求问题，比如：第二台机器很慢，但没挂，当请求调到第二台时就卡在那，久而久之，所有请求到第二台机器的请求都卡在第二台上，不影响其他线路。

  • 解决办法 :结合权重，把第二台机(性能低的)的权重设置低一点

Least Active Load Balance

  • 相比于随机负载均衡，最小活跃多了几个属性

  • 一个长度为invokers.size()的for循环判断出最小活跃数的个数，以及最小活跃数的下标

  • 断言least Count大于0assert(leastCount > 0) ,则往下走(此处默认大于0)

  • 如果只有一个最小,即leastCount == 1 ,则直接返回return invokers.get(leastIndexs[0]);

  • 否则存在多个最小活跃数,此时判断每个最小活跃权重是否相同，如果不同且总权重大于0，则按总权重数随机

    return invokers.get(leastIndex);

  • 否则均等随机(权重相同或权重为0) return invokers.get(leastIndexs[random.nextInt(leastCount)]);

Consistent Hash Load Balance

  • 判断当前的请求参数之前是否已经被请求过
  • 如果没有请求过则创建一个新的请求
  • 否则直接return到之前请求过的对应provider

ps:当某一台提供者挂时，原本发往该提供者的请求，基于虚拟节点，平摊到其它提供者，不会引起剧烈变动。
